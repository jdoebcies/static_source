高效可观测性数据存储的六项功能要求

高基数高维度

任何事件中的任何字段都必须可查询，不能被预先聚合，因为你事先不知道哪些字段相关

维度平等

不能把任何特定维度的检索性能置于其他维度之上，所有可能需要的维度必须同样快

时间特权

时间维度是唯一例外的特权数据：查询总是针对特定时间范围，必须适当索引

近乎实时

从系统接收数据到可查询不应超过几秒钟，陈旧数据导致工程师在次要线索上浪费时间

持久可靠

关键调查中不能丢失可观测性数据，存储持久性必须经受住基础设施内部故障

容错

检索机制必须容错，任何底层组件故障下仍能快速返回查询结果

为什么时序数据库不能胜任可观测性

设计初衷

TSDB 为聚合指标设计：创建新时间序列开销大，但为已有时间序列添加数值开销低

高基数冲突

将每个维度转为标签、每个值转为标签值，会为每个独特组合创建新时间序列，导致时间线爆炸

根本矛盾

结构化事件每个都唯一，建行开销与事件数线性增长，建造成本无法在多次测量中摊销

存储策略 · 双张力
06 / 22
行存储

检索单行快，主键索引按时间

但任意字段分析需要读取该行 tablet 的所有列，丢弃大部分数据。Bigtable 的列簇 locality group 也要求读取整个 group，索引开销大到不可接受（Dapper 为三个字段建索引产生数据量是追踪数据本身的 76%）

列存储

独立访问每列数据，按需扫描

但行数据没有顺序保证，全行扫描需要拉出所有列文件。Dremel/ColumnIO 用手动分片分解表，碎片管理在大数据量下变得极其困难

段落拆解 · Bigtable
07 / 22

Bigtable 的压缩过程带来了性能窘境

Fact

Bigtable 通过内存 mutation 日志和覆盖文件栈管理更新，定期将叠加文件压实到基本不可变层，按记录先后顺序重写，磁盘 I/O 开销昂贵

Point

可观测性是一写多读：新数据必须几秒内可查，压缩要么持续运行要么读每个堆叠层。即使 locality group 也只允许检查组内列，仍需读取整个 group

Aftertaste

Dapper 后端为三个字段（服务、主机、时间戳）建索引，产生的数据是追踪数据本身的 76%。如果完全消除索引需求，就能得到更实用的方法

Retriever · 时间分区

按时间分区：仅追加的段模式，读取时纠正无序到达

写入策略

新 span 插入当前活动段文件末尾，追踪最早和最晚事件时间戳，创建可能与其他段窗口重叠的窗口

段最终化

一小时过去、25 万条记录或 1GB 数据后，段被最终化为只读，记录其最旧和最新时间戳到元数据

查询过滤

查询时只获取与查询时间窗口重叠的段，不相关片段的数据完全不被检查

Retriever · 列式布局
11 / 22

每段每字段一个追加文件，按行序列号重建原始行

Fact

事件分解为组成字段，每个段每个字段只有一个追加文件，加上特殊的时间戳索引字段。每行分配时间戳和段内相对序列

Point

查询时只访问相关列文件。每列文件在磁盘上交替存放序列号和列值，通过访问对应列文件重建原始数据行

Aftertaste

列创建成本在写入该列的所有值中摊销。偶尔有非空值时创建新列性能良好，一次性使用的列性能不佳（不要创建名为 timestamp_2021061712345 的列）

权衡 · 实时与成本
15 / 22
实时可查

开放列文件总是可查询，采集与查询完全分离

查询过程可强制冲刷部分文件，即使尚未最终化和压缩。查询处理器完全通过磁盘文件操作，采集专注创建文件，不必维护共享状态

存储分层

旧数据卸载到 S3 等持久存储，按需取回

最近几分钟在本地 SSD 上，超过一定年限的封闭段压缩上传至 S3。查询需要时从 S3 取回到内存中解压并扫描

无服务器 mapper-reducer 模式：并行且快速

Fact

每个段的结果可独立计算，不需要顺序处理。云提供商以分布式方式存储文件，没有单点拥堵

Point

Lambda 函数作为 mapper 独立处理段目录，结果通过中间合并集合提供给集中 reduce worker 计算最终结果

Aftertaste

不耐烦策略：90% 请求完成后重新请求剩余 10%，竞争最先返回者。代价高 10%，但避免了被 S3 阻塞的查询

处理高基数分组的三层策略

哈希环分布

遵循 Chord 模式，创建组的哈希值并在覆盖键空间的环中查找对应关系，使 reduce worker 处理均衡子集

内存限制

通过所有 reducer 可用内存总量限制返回组数，估计哪些组最可能在用户提供的 ORDER BY / LIMIT 标准下存活

实用主义

十万条独特线条在 2000 像素高的图表上毫无意义，干脆放弃返回太多组的查询

Kafka 流式数据模式：分离采集与序列化

接收器

轻量无状态进程验证入站行，将每行生成到选定 Kafka 主题和分区中

索引工作者

有状态进程从每个 Kafka 分区按顺序消费数据，序列化为段文件和目录

核心总结
21 / 22
01

可观测性数据存储有六项硬要求：高基数高维度任意字段可查、维度平等不偏袒、时间特权索引、近乎实时新鲜度、持久可靠、容错。这些要求共同塑造了存储方案的选择空间

02

TSDB 为聚合指标设计，面对高基数结构化事件会发生时间线爆炸：每个独特标签组合创建新时间序列，建行开销与事件数线性增长，无法在多次测量中摊销

03

纯行存储（Bigtable）检索单行快，但任意字段分析需读全列。Dapper 为三个字段建索引，产生数据量是追踪数据本身的 76%。纯列存储（Dremel）独立访问列但全行扫描需遍历所有列文件。两者都不能单独满足可观测性

04

Honeycomb Retriever 的混合方案：按时间分区的仅追加段模式（段内无序写入，读取时按段元数据过滤），段内按列存储（每字段一个追加文件），同时获得行和列的部分扫描能力

核心总结
22 / 22
05

查询流水线六步：匹配段→列扫描→行重构→段内聚合→跨段聚合→Top-K 排序。每行处理完立即丢弃，RAM 只存一个值而非整个段文件。无索引的情况下处理任意过滤器组合

06

压缩层层递进：字典编码、稀疏编码、运行长度编码减少列内冗余，再用 LZ4 整体压缩段。列创建成本在写入该列的所有值中摊销，偶尔非空值性能良好，一次性列性能不佳

07

并行加速采用无服务器 mapper-reducer 模式 + 不耐烦策略：每个 Lambda 独立处理段目录，reduce worker 合并结果。90% 完成后立即重请求剩余 10%，竞争最快返回者，代价高 10% 但避免阻塞

08

Kafka 流式分离采集与查询：轻量接收器验证并生成至分区，有状态索引工作者按序消费序列化。查询与采集只共享文件系统不共享 RAM，容错隔离。截至 2021 年集群：700 TB 数据、5 亿段、150 万 spans/s、中位延迟 50ms、p99 5s
