复杂度要分清两种来源

必要复杂度

给定情况固有的复杂度，无法从问题定义中移除。Web 服务器必须处理快速提供页面的问题。

意外复杂度

不固定，可以通过工程努力解决。用 Java 写服务器时减少 GC 影响，可能引入这类复杂度。

明确的、最小的 API 是管理简单性的必要部分

提供给消费者的方法和参数越少，API 就越容易理解，团队也越能完善这些方法。

材料把有意识地不解决某些问题，视为专注核心问题的方式。很小、很简单的 API 通常也是深刻理解问题的标志。

compatibility as simplification14 / 18
release simplicityp133

发布流程越简单，变化影响越容易被理解

大批量发布

如果同时发布 100 个不相关的更改，性能变差后要花大量时间定位原因。

小批次发布

每个变更的影响可以独立理解，因此可以更有信心地进行更快发布。

smaller batches15 / 18
小批次发布像梯度下降法，每次进展一点。

每次改变都要观察对系统的改善和退化，再继续寻找最佳方案。

chapter summaryp133 p134

软件的简单性是可靠性的前提条件

考虑如何简化任务的每一步，不是在偷懒，而是在明确实际要完成什么，以及如何更容易做到。

对新功能说不，是保持环境整洁，避免分心，让团队持续关注创新和真正的工程工作。

核心总结

01

生产软件持续变化，SRE 要在稳定性和灵活性之间维持平衡。

02

乏味、可预见、缺少意外，是源代码和生产环境的理想状态。

03

新增代码、新 API、新发布批次都会增加风险，删除和拒绝同样是工程动作。

04

模块化、兼容数据格式和小批次发布，都是把系统变化压到可理解范围内。
